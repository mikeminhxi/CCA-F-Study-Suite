Traps to reject
  • Chain-of-thought can't fix a missing-context problem — the model can't reason about code it never sees.
  • Recursive / nested agent hierarchies and external message queues wreck observability — the exam always prefers a flat coordinator.
  • Automatic context inheritance between subagents does not exist — the coordinator must pass everything explicitly.
  • Adding emphatic prompt instructions asking for parallelism doesn't create it; the structural mechanism does.

Prompt Engineering & Structured Output

20%
Decision rules
Behaviour/tone must persist across all interactions
→
System prompt
Inject a live external status update for the next reply
→
Add it to the system prompt before the next call
Repetitive openers ("Certainly!", "I'd be happy to…")
→
Prefill a partial assistant message
Inconsistent format ("cotton blend" vs "Cotton/Poly mix")
→
Few-shot examples of the canonical format
Model invents values for fields not stated in source
→
Instruct it to return null when info isn't present
Recall misses because info appears in varied layouts
→
Few-shot across the format distribution
Enum keeps failing as new categories appear
→
Add "other" + a detail string field
Invoice line items don't sum to the stated total
→
Capture calculated_total AND stated_total, flag mismatches
Semantic errors slip past JSON schema validation
→
Field-level confidence + calibrated review thresholds
Dependent tools; one must run first (needs the other's output)
→
tool_choice = that specific tool on turn 1, then back to auto
Ambiguous request; too many clarifying questions cause drop-off
→
State assumptions & proceed; save questions for irreversible actions
Traps to reject
  • Making fields required to stop nulls — that forces the model to invent values.
  • Temperature = 0 to fix format/hallucination — determinism just makes the wrong answer consistent.
  • Upgrading the model tier as a first move — the exam wants you to fix the prompt/schema.
  • tool_choice="any" for ordering — forces a tool, not the right one; and "reorder the array" relies on undocumented behaviour.
Keyword triggers
persist across all = system prompt repetitive openers = prefill canonical format = few-shot not stated in source = return null new categories appear = enum + other must run first = tool_choice specific
DOMAIN 2

Tool Design & MCP Integration

18%
Decision rules
Tool fails; how to signal the error to the agent
→
Error text in content with isError: true
Agent retries permanent errors, escalates transient ones
→
Structured error metadata: category, retryable bool, cause
Agent ignores your MCP tool, uses Grep/Write/sed instead
→
Improve the tool description (capabilities, outputs, when to use)
Standard third-party service data (Jira tickets)
→
Integrate an existing MCP server
Exploratory calls, no visibility into server content, context exhaustion
→
Expose content as MCP resources
One generic free-text tool gives ambiguous output
→
Split into purpose-specific tools with I/O contracts
Semantically overlapping tools cause mis-selection
→
Consolidate/merge with an action parameter
Agent picks wrong tool among 18
→
Give each agent a small, role-relevant tool set (4–5)
Agent omits required params / mis-structures data
→
Clear parameter descriptions with format hints
Wrong entity picked (nicknames, rematches, formats)
→
Replace params with an id + a separate search/lookup tool
Model proceeds on low-confidence / over-asks for review
→
Return a requires_review bool from tested thresholds + reasons
How does the agent reach tools across 3 MCP servers?
→
All discovered at connection, available simultaneously
Traps to reject
  • Returning empty results or throwing exceptions on tool failure — breaks the protocol; use isError + message.
  • Removing the competing tool or adding prompt routing rules to force MCP adoption — fix the description instead.
  • Building a custom MCP server / curl in Bash for standard service data — an existing server is the answer.
  • An extra analyze_error tool — put the metadata in the original response; don't add a round-trip.
Keyword triggers
Operation failed / uniform errors = structured metadata agent won't use the tool = better description Jira / standard service = existing MCP server no visibility / exploratory = MCP resources overlapping tools = consolidate
DOMAIN 5

Context Management & Reliability

15%
Decision rules
Agent re-asks for info given earlier in the chat
→
API is stateless — pass the full messages array each request
Returning customer; old tool results now stale
→
New session + structured summary + fresh tool calls
Multi-issue conversation nearing the context limit
→
Progressive summarization: compress resolved, keep active verbatim
Story/persistent facts must stay consistent over 40+ turns
→
"Story bible" retained section; summarize only the ephemeral part
Tool outputs (40+ fields) bloating the context
→
Prune to task-relevant fields (not a vector DB, not prose)
Non-blocking work, want the 50% discount
→
Batch API — suitability = results useful up to 24h later
Batch window vs SLA math
→
max wait + 24h ≤ SLA, leave a cushion
3% of batch failed with context_length_exceeded
→
Chunk only the failures, extract per chunk, merge
Rule "cannot be left to model discretion" (e.g. $500 refund)
→
Deterministic hook outside the model
When should the agent escalate to a human?
→
Natural-language criteria: asks for human / policy exception / no progress
Escalating mid-case to a human without your transcript
→
Structured handoff: id, root cause, amount, recommended action
Backend fails but you can still explain / verify
→
Graceful degradation: give partial value, be honest, offer options
Traps to reject
  • Emphatic prompts / few-shot for a compliance rule — "cannot be left to model discretion" rules out any prompt-only fix.
  • Sliding window that silently drops the very issue the user later asks about.
  • max_tokens to fix context_length_exceeded — that's output length, not input.
  • Reprocessing the whole batch to fix a 3% failure — target only the failures.
  • Dumping the full transcript on escalation — the human wants a one-screen brief.
Keyword triggers
asks for name again = stateless, pass history 50% discount / 24 hours = Batch API cannot be left to discretion = hook context_length_exceeded = chunk failures human won't see transcript = structured handoff

04

How to Run Your Deep Dive

You said you'll deep-dive next. Here's a sequence that spends your time where the marks are and turns recognition into recall.

STEP 01

Skim by weight

Read D1 and D3/D4 decks first. They're 67% of the exam. Don't open a single practice question yet — learn the rules cold.

STEP 02

Attach rules to questions

Now work the 157 Qs. For each, name the trigger phrase before reading the options. If you can't, that rule needs another pass.

STEP 03

Mine the traps

Every wrong answer you pick maps to a trap in these decks. Log which trap fooled you — patterns repeat far more than facts.

STEP 04

Decoder-only review

Final day: section 03 alone. Filter by domain, cover the right column, recite the answer. When it's reflex, you're ready.
